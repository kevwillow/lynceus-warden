{# Reusable data-table foundation for the web UI's server-rendered, server-paginated tables. Two macros, both pure presentation -- the route owns the query (filters, sort, pagination); these own the wrapper + header + footer chrome that every list page repeats. data_table(columns, endpoint, filter_qs, sort, dir, table_id=none) Renders the .table-scroll wrapper, a of sort-aware headers, and the ; the caller supplies the row markup as the {% call %} body, which lands inside . ``columns`` is a list of dicts: {'label': str, 'sort': key?, 'nowrap': bool?}. A column carrying a 'sort' key renders a plain GET that carries the current filters (``filter_qs``) and toggles direction on the active column; a column without 'sort' renders a plain, non-clickable header. Sort links deliberately omit ?page -- a new ordering invalidates the page position, so the route resets to page 1. ``nowrap`` emits an inline ``white-space: nowrap`` on the : .table-scroll's global th/td nowrap already covers this, but the flag preserves a column's belt-and-suspenders inline declaration (e.g. the /devices Type header, hardened after a real truncation bug) so a future CSS override can't silently re-introduce a squash. pagination_footer(endpoint, base_qs, page, total_pages, total, per_page) Renders the prev / page-indicator / next nav. Serves BOTH footer idioms in the codebase from one signature: callers on the "individual vars" idiom (devices/probes) pass only page + total_pages and get the minimal "page X of Y"; callers on the "pagination object" idiom (alerts/watchlist/watchful/allowlist) also pass total + per_page to append " · N total · per_page=K". ``base_qs`` is the current filter + sort state without ?page, so paging preserves both. has_prev/has_next reduce to page > 1 and page < total_pages, which hold for either idiom. #} {% macro data_table(columns, endpoint, filter_qs, sort, dir, table_id=none, resize=false, hide=false) %} {# Per-feature opt-in (v0.9.2). ``table_id`` names the table (its localStorage key + data-table-id value); ``resize`` and ``hide`` opt it into, respectively, the client-side column resize/reorder grips and the show/hide columns menu. Sorting is NOT a flag: each header's sort is emitted from that column's own 'sort' key (server-side, route-owned) independent of this layer, and the 5th macro arg ``sort`` is the active sort key, not a toggle.
